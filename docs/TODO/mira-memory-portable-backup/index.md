# Mira 记忆与聊天可移植备份

## 原因

旧“记忆库导出”只覆盖 ObjectBox 通用记忆。新版角色记忆、证据、事件、关系边和角色授权位于 Room，用户需要额外理解并操作数据库备份，界面含义与实际导出范围不一致。聊天归档也未持久化情感朗读方向。

## 修改

- 记忆导出统一为 `mira_memory_archive` JSON
- 同一文件包含旧记忆、结构化角色记忆、证据、事件、关系边和授权
- 导入支持跳过、更新、始终新建，并继续接受旧 `memory_backup_*.json`
- 跨配置导入重写记忆归属配置，结构化记录统一绑定目标配置
- 结构化记忆 ID 变化时同步重写版本链、证据、关系边与授权引用
- 普通证据和关系边证据都按会话与时间戳重新绑定，避免跨设备复用错误的数据库自增 ID
- 统一记忆包格式升级到版本 2，新增 `edgeEvidence` 可移植关系证据；版本 1 和旧记忆 JSON 继续可导入
- 聊天与记忆可按任意顺序导入；聊天导入完成后会自动重绑尚未解析的普通证据和关系证据
- 聊天归档格式升级到版本 3，Room 数据库升级到版本 28，保存情感朗读方向和待重绑关系证据
- 新导出的聊天归档标识为 `mira_chat_archive`，旧 `operit_chat_archive` 文件仍可导入
- 聊天 JSON 在写入前执行全文件校验，整批写入使用 Room 事务，坏包不会只导入前半段
- 旧 ObjectBox 记忆导入使用事务，更新模式完整恢复创建时间、更新时间、标签和属性
- Room 写入失败时按导入前 ObjectBox 快照执行补偿回滚，避免正常异常路径留下半套统一记忆
- 旧数据库从聊天角色名推导出的 `character_name:*` 会在角色卡可解析时事务合并到稳定 `character:*` ID，避免迁移后记忆存在却检索不到
- 默认 Mira 会话即使未显式写入角色卡名称，也会绑定稳定默认角色 ID，并收回此前因空角色 ID无法召回的会话记忆
- 旧关系边只剩失效本机消息 ID 且已经无法反推出会话出处时，保留关系并写入明确日志，不伪装为完整证据
- 备份页统计和结果信息覆盖新版记忆数据

## 验收

- 导出文件同时存在 `legacyMemory` 和 `structuredMemory`
- 结构化部分包括 `records`、`evidence`、`episodes`、`edges`、`edgeEvidence`、`grants`
- 旧记忆 JSON 仍可导入
- 跨配置导入后记忆属于目标配置
- 恢复聊天后情感朗读方向仍存在
- Room 26 升级到 28 时保留已有聊天与记忆
