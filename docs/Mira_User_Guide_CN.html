真机新手流程：从干净安装到插件调用

这一章按照一台真实 Android 手机的操作顺序编写。不要跳着配：先跑通文字模型，再配语音、终端和 Shizuku，最后安装并验证插件。

密钥只在官方平台创建，只填进 Mira 的本地配置 不要把完整 Key 发到聊天、群聊、Issue、截图或日志里。教程中的密钥一律写成 YOUR_API_KEY 或使用掩码截图。

A. 干净安装和首次引导

  1. 安装 APK，阅读本地数据与云端服务说明并同意。
  2. 按引导完成通知、存储和基础权限。麦克风、悬浮窗、位置、Shizuku 可以等真正使用时再开。
  3. 填写用户称呼和偏好。这里是用户资料，不是角色卡；以后可在设置中修改。
  4. 进入聊天首页后，先确认顶部角色名、模型入口和输入框都能正常显示。

B. 配置 DeepSeek 主模型

设置模型与 API新增服务DeepSeek
  1. 新手通常没有 API Key。先在 DeepSeek 官方开放平台完成注册、实名与充值，再创建 Key。
  2. Key 只会完整显示一次，立即保存到密码管理器；截图时必须掩码。
  3. 回到 Mira，填写官方 Endpoint、API Key 和模型 ID，执行“测试连接”。
  4. 在聊天页发一句普通文本。必须同时验证：用户消息已保存、模型开始流式输出、回复结束后请求状态恢复。
Provider: DeepSeek
Endpoint: https://api.deepseek.com
API Key: YOUR_API_KEY
Model: 以官方控制台当前可用模型为准
发送后不回复时先看状态依次检查连接测试、Endpoint、模型 ID、余额、网络、401/404/429/超时错误；不要先去改角色提示词。

C. 配置豆包 Seed-TTS 2.0

设置语音与通话TTS豆包 Seed-TTS 2.0
  1. 在火山引擎控制台进入“系统管理 → API Key 管理”，创建或复制 Token。
  2. 新版接入只填写 Token；旧版 App ID 字段留空。不要把 LLM Key 填到 TTS。
  3. 选择音色，例如 Vivi 2.0；搜索框应能正常唤起键盘。
  4. 先用“试听”验证，再为当前角色开启自动朗读。自动朗读不等于实时语音通话。
语音验收文本必须先可见，语音随后播放；用户再次发送消息时应停止上一段朗读；正在朗读的高亮只标当前句。

E. 补齐悬浮窗、电池豁免和位置权限

设置高级功能权限授予
  1. 悬浮窗用于陪伴气泡和快速回复。
  2. 电池优化豁免用于后台陪伴、提醒和悬浮服务保活。
  3. 位置权限只在天气、位置相关工具或主动陪伴规则需要时开启。
  4. 授权后回到权限页刷新，不能只相信系统弹窗已经点过。

F. 配置 Shizuku 调试权限

  1. 安装并打开 Shizuku，选择通过无线调试启动。
  2. 系统设置中开启开发者选项和无线调试，选择“使用配对码配对设备”。
  3. 六位配对码只在本机通知栏输入，不截图、不发送给别人。
  4. 配对完成后返回 Shizuku 点“启动”，确认显示“Shizuku 正在运行”。
  5. 回 Mira 的“权限授予”，点红色“Shizuku 服务 / 未授权”，展开后授予权限，并设为当前级别。

G. 新建角色卡并开始独立会话

聊天页顶部角色名创建 / 管理角色群聊角色卡+
  1. 顶部显示的是当前角色名，不要把教程写死成 Zero。示例设备的默认角色叫 Mira。
  2. 填写角色卡名称、描述、角色设定、开场白、“其他内容（聊天）”和“其他内容（语音）”。
  3. 新建表单若继承了默认 Mira 内容，必须全选替换，否则只是换名字。
  4. 点击“+ 添加”选择公开头像；相册页不进教程截图。
  5. 保存后点角色卡右上角三点，选择“设为活跃”。卡片主体目前不会切换角色。
  6. 新建对话。旧消息仍显示原角色署名是正确行为；新会话才绑定新角色。

H. 用自然语言调用终端工具

用户不需要自己逐条敲命令。直接在聊天中说明“调用终端工具执行什么，并保留原始输出”。例如：

请调用终端工具执行：
date "+%Y-%m-%d %H:%M:%S %Z" &&
python3 --version &&
node --version &&
pnpm --version。
请保留完整工具执行轨迹，并把原始输出整理给我。

验收时要看到执行过程、真实命令和原始输出。Markdown 表格列没显示全时，在表格内部左右滑动；这不是结果丢失。

I. 从市场安装并启用插件

设置高级功能扩展与技能黄色小店图标
  1. 市场搜索目前可能返回无关条目；找不到时回首页按分类和最近更新浏览，不要反复猜关键词。
  2. 打开插件详情，检查作者、版本、简介和是否需要额外账号或 Key。
  3. 点“安装插件”只完成下载与解包。

J. 聊天、记忆与隐私验收

  • 聊天记录应保存在本地会话数据库，重启应用后仍能恢复。
  • 会话抽屉应支持搜索历史聊天；新建对话时有草稿则选择“保存草稿并新建”。
  • 角色、关系记忆和会话摘要必须按角色与会话隔离。
  • 导出和备份前检查文件大小，备份副本不要只留在同一台手机。
  • 公开截图前移除 Key、Token、手机号、验证码、支付、位置、Wi-Fi/BSSID、私人聊天和文件路径。

1. 先弄懂 Mira

Mira 是一款 Android 私人 AI 伴侣。它把聊天、角色、长期记忆和语音放在前台，同时保留文件、终端、Skill、MCP、工作流与设备协助能力。

一句话理解工作方式

聊天记录

默认保存在手机本地，用来恢复会话。

长期记忆

在本地独立保存，可查看、纠正和删除。

AI 回答

请求直达你选择的模型服务商。

主动消息

由手机本地任务触发，不依赖 Mira 自建服务器。

“本地优先”不等于“所有推理都离线” 角色、记忆、聊天记录和主动陪伴逻辑可以留在手机上；如果选择云端模型或云端 TTS/STT，请求内容仍会发送给对应服务商。是否收费、是否保存日志，由该服务商的规则决定。

最少需要准备什么

  • Android 8.0 或更高版本的手机。
  • 一个可用的对话模型 API Key，或已经配置好的本地模型。
  • 稳定网络；使用海外服务商时还要保证手机能连接其接口。
  • 想用语音时，额外准备 TTS/STT 服务，或使用系统/本地语音服务。

新手推荐路线

  1. 先只配置一个对话模型，完成一次普通文字聊天。
  2. 再设置名字、人设和自动记忆。
  3. 然后配置 TTS，确认能试听，再开启自动朗读和语音通话。
  4. 最后再碰 Skill、MCP、Shizuku、终端等高级能力。

Mira 与 Operit 的区别

Mira 不是换皮版 Operit。它保留 Operit 的 Agent、终端、Skill、MCP 和设备控制基础，但把默认产品路径从“先选工具”改成“先和一个能记住你的角色说话”。

维度Operit 基础Mira 的处理
第一入口Agent、工作区、工具与自动化角色聊天、关系、语音与主动陪伴；工具在任务需要时出现
聊天界面承载大量能力入口顶部只管会话和角色，底部只管输入与本轮能力，复杂设置收进抽屉
记忆记忆、知识库和图谱基础独立的伴侣记忆库，区分用户、角色、关系、会话，并保存证据、版本与冲突历史
语音多供应商 TTS/STT情绪分段、播放队列、当前句高亮、新消息打断，并保证文字先显示
主动性定时任务和工作流结合角色、最近会话、约定与记忆生成主动消息
插件OperitForge 与现有协议Mira 市场和 MiraForge 优先发布，同时继续读取 Operit 兼容资源
发布Operit 包名、仓库和更新源独立应用 ID、Mira 图标、Mira GitHub Releases 更新源
兼容是底层能力，不是产品主界面 旧插件、Skill、ToolPkg 和部分内部 com.ai.assistance.operit 标识会继续保留，避免破坏协议、数据库迁移和已安装扩展。Mira 的源码、Issue、Release 与更新通道仍由自己的仓库维护。

4. 主界面地图

手机聊天页没有常驻底部导航。顶部管理会话和角色，中间只放消息，底部只保留会话控制条和输入框；记忆、设置从左上角会话抽屉进入。平板和大屏可常驻会话侧栏，但聊天正文仍限制最大宽度。

  1. 左上菜单：打开会话列表，搜索、切换或管理历史对话。
  2. 角色区域：显示当前角色；点击可切换角色或群组。
  3. 新建对话：右上角带加号的对话图标，立即创建空白会话。
  4. 更多菜单：当前聊天的导出、设置和高级操作。
  5. 会话控制条：位于输入框上方，用于附件、模型、回复模式和会话能力。
  6. 输入区：只负责文本、语音、发送和停止生成。
  7. 发送 / 语音：输入框有文字时显示发送；空白时可进入语音输入。
会话抽屉

新建、搜索、最近、归档、记忆和设置。

角色菜单

切换角色、查看资料、关系状态和主动陪伴。

输入框上面的控制条是什么

  • 加号：打开轻量附件菜单，只保留相机、照片、文件和引用记忆。
  • 模型名称：当前会话绑定的模型配置；点击可快速切换。
  • 回复模式：选择轻快、智能或深入；窄屏会与模型名称合并显示。
  • 会话能力：默认允许模型按需调用工具；附件和设备协助按当前权限生效。
  • 灰色接话建议：输入框为空时，功能模型根据上一轮对话生成的候选回复；点击输入框开始打字时会消失。

5. AI 模型配置

模型配置决定“请求发到哪里、用哪个模型、允许生成多长”。API Key 像密码，Endpoint 像地址，模型名称像收件人。

会话抽屉设置模型与 API模型参数

四个必懂字段

字段含义小白常见错误
API 提供商接口协议类型，例如 OpenAI、Anthropic、Gemini、DeepSeek、豆包只看模型名字，选错协议
API Key服务商控制台创建的密钥复制了前后空格，或把账号密码当 Key
Endpoint / API 地址请求的完整接口地址或基础地址漏掉 https://，或多写/少写路径
模型名称服务商实际支持的模型 ID填了宣传名，而不是 API 模型 ID

通用配置步骤

  1. 点“新增配置”，先起一个容易认的配置名，例如“DeepSeek 日常”或“豆包陪伴”。
  2. 选择真实供应商；兼容 OpenAI 协议的中转服务通常选择“OpenAI 通用”。
  3. 粘贴 API Key，填写服务商文档给出的 Endpoint 和模型 ID。
  4. 按模型能力开启识图、音频、视频或 Tool Call，不要给纯文本模型乱开多模态。
  5. 先执行“测试连接”，至少测试普通聊天；要让 Mira 操作工具，再测试 Tool Call。
  6. 保存后回到聊天页，通过输入框下方的模型按钮选择它。
AutoGLM 不适合作为日常对话主模型UI 控制模型和聊天模型是两件事。日常聊天请选择正常对话模型；AutoGLM 只在明确要求智能体控制界面时使用。

上下文和生成长度怎么设

  • 上下文长度：Mira 日常实际使用的聊天窗口。新手先用模型支持范围内的中等值。
  • 最大上下文长度：供应商文档声明的硬上限，不要凭感觉写得比模型能力更大。
  • 最大生成 Token：单次回答最长能输出多少；太小会截断，太大可能变慢和变贵。
  • 总结阈值：上下文使用到一定比例后，Mira 会压缩旧对话，最近消息仍保留。
推荐原则先用默认参数跑通，再一次只改一个参数。出现截断就调生成长度，出现超限就检查上下文上限和总结阈值。

思考模式和档位

思考模式适合复杂规划、代码、推理和工具任务。普通闲聊关闭会更快、更省钱。档位越高通常越慢、Token 越多；是否真正生效还取决于供应商和模型是否支持对应参数。

6. 功能模型配置

主模型负责和你聊天；功能模型负责标题、记忆、总结、接话建议、图片识别等后台小任务。分开配置能兼顾质量、速度和费用。

UI 控制器专用视觉控制模型只用于设备界面自动化
群组规划速度与规划能力平衡的模型决定多角色发言顺序
“自动记忆没反应”先查这里确认“记忆更新”已经绑定可用模型，并且该配置测试连接成功。功能模型失败不会总是影响主聊天，所以很容易被忽略。

7. 聊天完整用法

新建、切换和搜索对话

  • 右上角“对话加号”直接新建会话。
  • 左上角菜单打开会话抽屉，可搜索标题和切换历史。
  • 每个会话可绑定自己的角色、模型、思考档位和上下文设置。
  • 删除前先确认是否需要导出；删除操作通常不可撤销。

发送附件

点会话控制条左侧加号，从轻量附件菜单选择相机、照片、文件或引用记忆。文件和图片是否能被理解，取决于对应功能模型和能力开关。

照片与文件

从相机、相册或文件选择器添加当前对话需要的内容。

引用记忆

从本地记忆中选择一条证据明确的内容加入本轮上下文。

判断附件是否真的带上了发送前输入区会出现附件提示；发送后消息详情中应能看到附件引用。图片“上传成功但 AI 看不懂”，通常是模型没有视觉能力。

消息长按菜单

不同消息可执行复制、编辑、删除、重新生成、回档、查看信息、切换回答候选和生成分享图等操作。重新生成会保留候选版本，可通过计数器前后切换。

当前触发的 Skill

当某个 Skill 被匹配并参与本轮处理时，聊天界面会显示当前 Skill。它表示本轮加载了哪套专业说明，不等于每次都执行设备操作。

智能接话建议

会话抽屉设置显示与行为智能接话建议

默认开启时，输入框为空会出现灰色建议。点击发送可直接采用；一旦你主动点输入框开始输入，建议会消失，不会混进你自己的文字。想提升速度，请在功能模型中给“智能接话建议”绑定快速模型。

8. 角色与人设

Mira 是产品名，默认角色只是一个可替换预设。Zero、Mira 或你导入的任何角色都不应该被写死在底层。

顶部角色名角色资料陪伴者设置

角色卡里建议写什么

  1. 身份：姓名、年龄设定、职业、生活环境。
  2. 性格：稳定特质和矛盾点，不要只堆“温柔、可爱”。
  3. 交流方式：用什么语言、句子长短、会不会用表情、如何称呼用户。
  4. 关系背景：双方认识多久、共同经历、当前关系。
  5. 边界和习惯：不喜欢什么、遇到不同意见怎么回应。
  6. 开场白：第一次见面或新会话的自然开场。
写人设的诀窍多写可观察行为，少写空泛形容。“用户难过时先听，再问一个具体问题”比“共情力很强”更容易稳定执行。

角色隔离

每个角色可以拥有独立头像、语气、模型、自动朗读设置和专属记忆。角色可选择使用专属记忆或跟随全局记忆，也可单独授权内建工具、包、Skill 和 MCP。

群组角色

群组模式允许多个角色参与同一会话。功能模型中的“群组规划”决定谁先说、谁跳过以及是否继续讨论。刚开始使用时建议先把单角色调稳定，再组群。

9. 本地记忆系统

聊天记录是“发生过什么”，长期记忆是“以后回答时值得继续带上的事实”。两者分开保存，避免每次把全部聊天原文塞给模型。

会话抽屉记忆

四种记忆范围

范围保存什么默认谁能读取
USER用户稳定身份、全局偏好和通用边界所有角色
COMPANION角色自身设定和角色私有认知仅该角色
RELATIONSHIP双方称呼、共同经历、约定和关系状态仅这一对用户与角色
CONVERSATION会话摘要、章节、当前话题和未完事项仅当前会话参与者

用户资料与角色共同经历不是一回事：前者可以全局使用，后者默认不跨角色。底层支持给其他角色授予或撤销单条记忆的读取权限；普通用户界面仍在完善时，以默认隔离规则为准。

Mira 会记哪些内容

身份与事实

名字、职业、家庭情况、稳定背景。

偏好与习惯

喜欢什么、讨厌什么、日常规律。

事件与约定

近期发生的事、承诺、纪念日和待办。

边界

明确不希望出现的话题或互动方式。

关系

双方称呼、关系阶段和共同经历。

会话摘要

长对话压缩后的关键背景。

自动记忆的流程

  1. 回复完成后，原始消息先保存，本轮再进入记忆候选队列。
  2. 记忆功能模型提出 CREATE / UPDATE / SUPERSEDE / LINK / IGNORE，并附带真实消息证据。
  3. 本地规则校验主体、范围、证据、重复、冲突和敏感内容，再决定写入或忽略。
  4. 写入后同步更新 Room 数据、全文索引和关系边；需要确认的内容进入“待确认”。
  5. 后续回答前，系统按角色、会话、相关性和 Token 预算召回当前有效记忆。
每轮判断，不等于每句话永久保存每个完成轮次都会进入候选处理，但闲聊、重复信息和低价值内容可以返回 IGNORE。明确说“请记住”会提高处理优先级。

证据、冲突与历史版本

  • 每条记忆可追溯到来源会话、消息、说话人、原始引用和时间。
  • 状态包含 ACTIVE、ARCHIVED、SUPERSEDED 和 DELETED。
  • 新事实替代旧事实时创建新版本，并用 SUPERSEDES 关系连接旧记录，不静默覆盖历史。
  • 默认回答只使用当前有效事实；查看历史或讨论过去时期时仍可追溯旧记录。

手动新增、确认和纠正

  • 新增：点“新增伴侣记忆”，选择归属、类型、主题和内容。
  • 确认：对“待确认”候选核对原话，确认后提高可信度。
  • 纠正：不要重复新建冲突事实，打开原记录点“纠正”。
  • 删除：会同时删除该记忆的证据、关联和检索索引。
归属选错会导致“她明明记住却不提”“当前会话”只在这一段聊天召回；“当前角色”跟着角色；“关于用户”的全局事实可跨角色使用。保存前先看归属。

关系档案与思维记忆网

默认“关系档案”适合普通用户，按关于我、关于角色、共同经历等方式浏览；“高级图谱”显示记忆节点和类别、范围、记忆域之间的关联，用于检查为什么某条记忆会被召回。

记忆没有自动保存时

  1. 检查功能模型中的“记忆更新”是否有可用配置。
  2. 确认这句话是稳定事实，而不是随口假设或临时玩笑。
  3. 到“待确认记忆”查看是否进入审核队列。
  4. 直接说“请记住：……”可提高意图明确度。
  5. 重要信息可手动保存，避免完全依赖自动提取。

10. 语音与通话

STT 把你的声音转成文字，TTS 把 Mira 的文字读出来。两者可使用不同服务商；语音通话只是把收听、发送、思考和朗读串成连续闭环。

会话抽屉设置角色与声音语音与通话

先配置 TTS，再开自动朗读

  1. 选择 TTS 类型：系统 TTS、硅基流动、MiniMax、MiMo、豆包、OpenAI、VITS 或 HTTP 自定义。
  2. 按服务商填写 API Key、完整 URL、模型和音色 ID。
  3. 打开“试听当前声线”，确认声音、速度和音高。
  4. 开启“情感朗读”，选择克制、自然或鲜明。
  5. 为当前角色选择“跟随全局 / 开启 / 关闭”自动朗读。

常见云端 TTS 字段

服务关键字段容易填错的地方
豆包 TTSAPI Key/Token、voice_type；旧版还需 App ID新版 API Key 与旧版 Access Token 混用
OpenAI HTTP TTS完整 /v1/audio/speech URL、Key、model、voice只填基础域名，漏掉接口路径
OpenAI RealtimeWebSocket URL、model、voice把 HTTP 地址填进 WebSocket 配置
硅基流动API Key、TTS 模型、预设或自定义音色自定义音色 URI 不完整
MiniMax / MiMoURL、API Key、模型、voice_id模型名使用了聊天模型而不是语音模型

配置 STT

本地 STT 更保护隐私但会占用设备资源；OpenAI、Deepgram 等远程 STT 依赖网络。远程 OpenAI STT 的 URL 必须完整包含 /v1/audio/transcriptions，模型可用 whisper-1 或服务商支持的转写模型。

语音通话怎么用

  1. 从聊天界面进入语音通话，允许麦克风权限。
  2. 看到“正在听”后自然说话，停顿达到设定时间会自动发送。
  3. Mira 思考并朗读回复；“允许语音打断”开启时，你可以在朗读中插话。
  4. 如果模型没有开始响应、整轮超时或发送失败，通话会恢复收听并给出状态。

推荐通话设置

  • 允许语音打断：默认开启。扬声器环境很吵、经常误打断时可关闭。
  • 停顿多久后发送：说话短促选“快速”；大多数人选“自然”；经常边想边说选“从容”。
  • 情感强度：先选“自然”。“鲜明”依赖 TTS 服务真正支持情绪或 style 参数。
  • 耳机：使用蓝牙耳机时保持系统媒体与通话路由正常，Mira 不强制切扬声器。
为什么调语速后听不出变化先确认当前 TTS Provider 支持 rate/pitch，并重新试听。部分服务只接受固定档位或忽略客户端参数；情感朗读还可能在基础语速上做小幅动态调整。

11. 沉浸对话

会话抽屉设置聊天与记忆沉浸对话

沉浸对话会把 AI 回复按句子拆成独立气泡，并按文本长度模拟发送节奏。它适合陪伴聊天，不适合需要一次复制整段代码或长报告的场景。

  • 可启用智能延迟，让短句先出现、长句稍后出现。
  • 可合并用户连续发送的短消息，停顿后统一交给 AI。
  • 可去除句末标点和显示标签，让聊天更接近即时通讯。
  • 工具能力仍可使用，但执行过程默认折叠，避免打断聊天氛围。
  • 可为当前角色配置表情包，支持 JPG、PNG、GIF 和 WebP。
调试问题时先临时关闭如果出现气泡顺序、延迟或内容截断问题，先切回普通模式确认模型原始回复是否完整，再判断是模型问题还是沉浸拆分问题。

12. 悬浮窗

悬浮球用于跨应用快速唤起 Mira；展开后是快速回复卡片，不会代替主应用里的完整聊天页。

会话抽屉设置主动陪伴悬浮陪伴
  1. 先授予系统“显示在其他应用上层”权限。
  2. 开启悬浮窗模式，屏幕边缘出现悬浮球。
  3. 拖动可横向和纵向移动；松手后可吸附左右安全边缘。
  4. 点击展开快速聊天卡片，可查看最近对话并发送消息。
  5. 点外部区域或关闭按钮收回悬浮球。
防止挡住系统按钮悬浮球会考虑状态栏、导航栏、刘海和安全边距。横屏或分屏后位置异常时，先收回再展开，让它重新计算可用区域。

悬浮窗看不到对话

  1. 确认主应用已经有当前会话，而不是停在空白未创建状态。
  2. 打开主应用切换到目标会话，再重新展开悬浮窗。
  3. 检查通知/前台服务是否被系统停止。
  4. 仍不同步时关闭悬浮窗服务后重新开启。

13. 主动陪伴与后台保活

Mira 的主动消息根据本地聊天上下文、记忆和提醒计划生成。没有 Mira 自建服务器替你保存关系档案，但 Android 是否允许它准时醒来仍受系统后台策略影响。

顶部角色名主动陪伴陪伴在线

建议开启的系统设置

  • 允许通知，并保留“Mira 在后台待着”的前台服务通知。
  • 把电池策略设为“不限制”或允许后台活动。
  • 在国产系统的自启动管理中允许 Mira 自启动。
  • 锁定最近任务，避免一键清理直接终止进程。
  • 需要准时提醒时，允许精确闹钟或日程权限（若系统要求）。
“纯本地”不代表永远准点息屏、超级省电、系统清理和厂商后台限制都可能延迟任务。重要闹钟仍应使用系统闹钟；Mira 的主动陪伴更适合问候、跟进和上下文提醒。

主动消息为什么能接上上下文

本地任务触发后，会读取当前角色、最近会话和相关长期记忆，再调用你配置的模型生成消息。主模型服务商仍会收到本次生成所需的上下文，但 Mira 不需要额外中转服务器。

14. Skill、MCP 与能力中心

Skill 是专业说明书，MCP 是标准化外部工具连接，工具包和工作流则提供实际能力。它们解决的问题不同。

会话抽屉设置隐私与高级功能高级功能
类型你可以把它理解成典型用途
Skill一份带目录和附件的专业操作指南志愿填报、写作流程、项目规范、领域知识
MCP让模型调用外部服务的标准插座搜索、数据库、GitHub、企业系统
工具包Mira 内可执行的一组工具文件、终端、网页、地图、自动化
工作流把多步操作固定成流程定期报告、批量处理、自动检查

Mira 市场与 Operit 兼容

Mira 默认读取自己的静态市场入口 https://kernelx30.github.io/Mira/market/v2/。新条目优先发布到 MiraForge；已有 OperitForge 条目继续按原来源下载和更新，不要求作者重新打包。

兼容不等于自动信任市场元数据来自 Mira，插件代码仍可能来自作者仓库或第三方服务。安装前检查作者、源码、Release、权限和网络端点。

打开 Mira 插件开发文档

安装一个 Skill

  1. 打开“高级功能 → Skill 市场”，或进入 Skill 管理器。
  2. 市场安装可搜索名称；仓库导入时粘贴 Git 仓库地址。
  3. 本地导入只支持合法 ZIP，包内必须能找到 SKILL.md。
  4. 安装后打开详情，确认入口文件、说明和附件都已识别。
  5. 进入角色的“工具访问”，允许该角色使用目标 Skill；同时确认内建 use_package 能力已开启。
  6. 回到聊天，用符合 Skill 描述的请求测试，观察聊天页显示的当前 Skill。

为什么装了却没触发

  • Skill 描述与当前问题不匹配。
  • 当前角色没有授权该 Skill。
  • 允许了 Skill/MCP，但没允许内建 use_package。
  • SKILL.md 的名称、描述或目录结构不规范。
  • 当前模型 Tool Call 能力不稳定，或模型配置未开启工具调用。

安装 MCP 前先看三件事

  1. 它需要哪些账号、Token、端口和运行环境。
  2. 它能访问哪些文件、数据库或网络服务。
  3. 它是本地进程还是远程服务，退出 Mira 后是否仍在运行。

15. 权限与设备控制

Mira 的聊天能力和设备控制能力是分开的。小白只聊天时，不必配置最高权限。

通道能力特点
普通 Android 权限通知、麦克风、位置、文件按功能逐项授权，最容易理解
无障碍读取界面结构、点击、输入和滚动兼容性广，但受页面结构影响
Shizuku通过系统服务执行更完整的设备操作通常需要无线调试配对，重启后可能要重启服务
Root最高级系统访问只适合清楚自己设备环境的用户

授权原则

  • 只开启当前任务需要的权限。
  • 给角色授权工具时，区分“能看到”和“能执行”。
  • 第一次自动化先用可撤销的小任务测试，例如打开设置页，不要直接做批量删除。
  • 系统升级或 Shizuku 重启后，重新检查连接状态。

16. 备份、迁移与隐私

会话抽屉设置聊天与记忆数据备份与恢复

建议备份哪些内容

  • 聊天记录和会话分组。
  • 伴侣记忆、证据和关系档案。
  • 角色卡、人设、头像与表情包。
  • 模型配置名称和参数；导出前确认是否包含 API Key。
  • Skill、MCP、工作流和自定义提示词。

聊天备份默认保存在 下载/Mira/backup。具体可导出内容以当前备份页面为准。

哪些内容会离开手机

功能可能发送的内容接收方
云端大模型当前消息、系统提示、选中的历史、相关记忆、附件和工具结果你配置的模型 Endpoint
云端 TTS / STT待朗读文本、音色参数，或录音片段对应语音服务商
搜索、MCP 与插件查询词、工具参数、选定文件或任务上下文对应网站、服务或插件端点
本地模型与本地语音推理本身可留在设备；联网工具仍按自身配置发送设备本地或工具配置的端点
备份离开应用沙箱后按敏感文件处理当前导出备份不要当作自动加密文件。它可能包含聊天、记忆、角色资料、Endpoint、Header 或 API Key；上传网盘、发给别人或附到 Issue 前先检查内容。

换手机的稳妥流程

  1. 旧手机先执行完整备份，并确认文件大小不是 0。
  2. 把备份文件复制到电脑或可信存储，不要只留在旧手机。
  3. 新手机安装相同或更新版本的 Mira。
  4. 先导入聊天与记忆，再检查角色卡和扩展。
  5. 重新填写敏感 API Key，重新授权麦克风、通知、悬浮窗和 Shizuku。
  6. 随机打开三个旧会话，并在记忆页搜索一条已知内容做验收。
分享日志、截图和配置前先脱敏检查 API Key、Bearer Token、聊天内容、文件路径、Cookie、地理位置和备份文件。清除 Cookie 只清理搜索工具、Browser 包和内置浏览器 Cookie，不等于清空全部聊天数据。

17. 外观与无障碍

会话抽屉设置隐私与高级功能主题和外观 / 显示与行为

Mira 使用无底栏的自适应聊天布局：手机保持单栏全屏聊天，平板和宽屏可常驻会话侧栏或信息侧栏；状态栏、系统手势区、键盘和刘海区域分别处理。

  • 主题和外观：颜色、明暗模式、聊天气泡和输入框样式。
  • 输入框悬浮：把贴底输入区改成悬浮圆角输入组件。
  • 显示与行为：时间显示、模型名、供应商、自动建议和其他全局行为。
  • 布局调整：用于极端字体大小、折叠屏或特殊分辨率下的边距微调。

大字体用户的建议

  1. 先保持系统字体在常用档位，观察聊天页和设置页是否完整换行。
  2. 不要同时把系统“字体大小”和“显示大小”都拉到极限后再判断单项问题。
  3. 出现按钮被挤出屏幕时，记录机型、分辨率、字体档位和具体页面截图。
  4. 横屏、分屏、折叠状态分别测试一次。

设置页怎么找东西

设置从左上角会话抽屉底部进入，按“角色与声音、聊天与记忆、主动陪伴、模型与 API、隐私与高级功能”分组。找不到时先判断它属于内容、模型、外观还是设备能力，不要在每个页面里来回翻。

18. 高级能力地图

Operit 官网把 Agent、自动化和开发能力放在主目录；Mira 仍保留这些底层能力，但统一收进“高级功能”，避免它们占据日常陪伴路径。先完成前面的聊天、语音和权限配置，再按任务开启。

与 Operit 官方教程的对应关系 Mira 已覆盖模型、终端、角色卡、语音、记忆、插件、权限、备份和常见故障；本章补齐本地模型、识图生图、工作流、深度搜索、UI 自动化、虚拟屏幕、工作区、外部调用以及角色卡分享。Operit 的桌宠和悬浮全屏工作台不作为独立主线：Mira 用陪伴气泡、快速回复卡片和正常聊天页承接。

识图、生图与功能模型

会话抽屉设置模型与 API功能模型配置
  • 识图：主模型或独立功能模型必须明确支持视觉输入；用一张无隐私的测试图验证内容、文字和位置描述。
  • 生图：在绘图工具包中配置供应商、模型和尺寸，再让聊天模型按需调用；文本聊天模型本身不会凭空生成图片文件。
  • OCR：属于设备或媒体能力，不放进默认附件菜单；需要时从高级功能或对应工具调用。
  • 音频/视频理解：分别绑定支持该模态的功能模型，别把 TTS 音色接口当成音频理解模型。

工作流、知识库与深度搜索

工作流

把多步工具调用固定成可复用流程，可由手动、定时或语音事件触发。

知识库

保存文档和项目资料；陪伴记忆仍走独立记忆系统，不与文件库混成一套。

深度搜索

组合网页、学术和地图检索，适合需要来源和多轮整理的任务。

  • 上下文总结：压缩长会话以控制 Token；总结可编辑，但原始消息仍应保留在本地聊天记录。
  • 工具箱：管理内置工具、工具包和执行权限；默认允许模型按任务调用，不再提供“仅聊天”模式。
  • 最小验收：先让一个工作流只执行两步，并检查每一步的输入、输出、失败状态和取消操作。

工作区、Web 开发与外部调用

会话抽屉设置隐私与高级功能工作区 / 开发者工具
  • 工作区：可绑定手机目录、SAF、SFTP 或 SSH，用于项目文件、代码编辑和 Git 操作。
  • Web 制作与打包：先在工作区生成并预览网页，再使用打包流程导出应用；预览成功不等于 APK 构建环境完整。
  • 终端开发：内置环境支持 Python、Node.js、pnpm、vim、SSH 和 Chroot；普通用户优先使用一键安装与自动验版本。
  • 外部调用：Tasker、广播或本地 HTTP 可触发指定代理/工作流；先用无敏感数据的固定文本验证入口，再接真实任务。
  • 开发辅助：返回码生成器、插件开发文档和调试工具服务开发者，不属于普通聊天必配项。

角色卡导入、标签、分享与群聊

  • 角色卡可从兼容 JSON、酒馆卡或项目支持的格式导入；导入后先检查人设、开场白、头像和模型绑定。
  • 标签用于管理大量角色和剧情线，不影响角色的真实记忆归属。
  • 导出或二维码分享前，移除私人聊天、用户资料、API Key 和不想公开的关系记忆。
  • 群聊中每个角色保留独立档案和说话方式，由群组规划模型决定发言顺序；用 @角色名 验证定向互动。
高级功能不是越多越好每次只开启一个新能力，完成一次可见验收再继续。工具调用失败时保留执行轨迹和错误详情，别只截最后一句“失败了”。

19. 故障排查

别一报错就乱改十个开关。按“现象 → 最可能原因 → 最小验证”排查。

发送消息后没有回复
  1. 到模型配置执行“测试连接”。
  2. 检查 API Key、Endpoint 和模型 ID。
  3. 确认手机能访问服务商域名。
  4. 切换到普通聊天、关闭思考模式，发送一句短文本。
  5. 查看错误详情是 401、404、429 还是超时：401 多为 Key，404 多为地址/模型，429 多为额度或限流。
输入框只有语音按钮，没有发送按钮
发送按钮只在存在可发送文本或附件时显示。灰色智能接话只是候选提示，不算正式输入；点候选采用，或开始键入自己的内容。
语音能识别，但 Mira 不朗读
  1. 打开“语音与通话”试听当前声线。
  2. 确认当前角色自动朗读不是“关闭”。
  3. 检查媒体音量和蓝牙输出设备。
  4. 云端 TTS 检查 API Key、完整 URL、模型和音色 ID。
语音通话把 Mira 自己的声音当成我说的话
优先使用耳机；保持“允许语音打断”开启以使用回声判断。如果设备扬声器回声严重，可暂时关闭打断、降低媒体音量或把停顿节奏调为“自然/从容”。
自动记忆不工作
检查“记忆更新”功能模型、待确认记忆、记忆归属和自动记忆开关。用明确句式“请记住：我的生日是……”做最小验证。
模型很快忘记前面的聊天
检查当前会话绑定的上下文长度、模型真实上限、总结阈值和是否出现 Token 超限提示。长期事实应进入记忆库，不要只依赖原始聊天窗口。
Skill 已安装但没有显示触发
检查 Skill 描述、角色授权、use_package、Tool Call 测试和 SKILL.md 结构。用与 Skill 描述高度一致的请求做测试。
悬浮球拖不动、位置不对或卡片空白
重新授权悬浮窗，关闭后再开启服务；主应用先切到目标会话；旋转屏幕后收回并重新展开。系统省电清理后需要重新启动悬浮服务。
主动消息不准时
允许通知、自启动、后台活动和不受限电池策略，保留前台服务通知。检查是否进入超级省电或被一键清理。主动陪伴受 Android 调度影响，不应用来替代关键闹钟。
应用闪退或界面卡住
记录发生时间、页面、操作步骤和机型；重新打开后导出错误报告与应用日志。反馈前删除 API Key、聊天内容和私人路径。若能稳定复现，写出“打开哪页 → 点什么 → 出现什么”。

一张排错顺序表

1. 只保留一个最小输入
2. 确认当前角色、会话和模型配置
3. 执行模型 / TTS / STT 的独立测试
4. 暂时关闭沉浸、思考、Skill 和自动化
5. 根据错误码检查 Key、地址、模型、额度和网络
6. 恢复一个功能，测试一次
7. 导出日志并记录可复现步骤
