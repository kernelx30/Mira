Mira 插件开发文档

从一句自然语言需求到可安装、可调试、可发布的插件。Mira 继续兼容 Operit 的脚本、ToolPkg、Skill 与 MCP 协议，新插件优先发布到 MiraForge。

1. 先选对扩展类型

类型适合什么主要文件运行位置
Skill给模型一套专业流程、规范和参考资料SKILL.md + 附件由模型阅读并按需调用工具
MCP连接已有的标准化服务或远程工具MCP server 配置或项目本地进程或远程服务
JS/TS 脚本实现一个轻量、独立的工具集合.js / .tsMira QuickJS sandbox
ToolPkg包含多个子工具、资源、UI、Hook、模板或 Providermanifest.json、main.js、子包与资源包级 main、sandbox、UI、provider 等上下文
别一上来就造 ToolPkg只有说明和流程时用 Skill；已有 MCP 服务时直接接 MCP；单个工具先写脚本。需要多个子包、UI、资源、生命周期或市场发布时再用 ToolPkg。

2. 应用内快速创作

普通用户不需要先学完整 API。Mira 已有“快速创作你的插件”入口：

  1. 进入“设置 -> 隐私与高级功能 -> 高级功能 -> 扩展与技能”。
  2. 打开“快速创作你的插件”，执行“自动下载并执行”，安装或更新包创作 Skill。
  3. 输入需求，例如“做一个批量整理下载目录图片并生成索引的工具”。
  4. 让 Mira 先生成目录、清单和脚本，再检查权限、网络地址、文件范围和错误处理。
  5. 安装后同时启用 ToolPkg 容器和目标子包，新建会话再测试工具调用。

这条路径会优先从 Mira 仓库下载创作支持，必要时读取 Operit 兼容源。最终产物仍需开发者自己检查，不能把“模型生成”当作代码审计。

3. 一个能跑的最小 ToolPkg

目录

hello_mira/
├── manifest.json
├── main.js
└── packages/
    └── hello.js

manifest.json

{
  "schema_version": 1,
  "toolpkg_id": "io.github.yourname.hello_mira",
  "version": "0.1.0",
  "author": ["YOUR_NAME"],
  "main": "main.js",
  "display_name": {
    "zh": "Mira 问候工具",
    "en": "Mira Greeting Tool"
  },
  "description": {
    "zh": "演示一个最小可调用工具",
    "en": "A minimal callable tool example"
  },
  "subpackages": [
    {
      "id": "hello",
      "entry": "packages/hello.js",
      "enabled_by_default": false,
      "display_name": { "zh": "问候", "en": "Greeting" },
      "description": { "zh": "生成一句问候", "en": "Generate a greeting" }
    }
  ]
}

main.js

function registerToolPkg() {
  // Register UI, hooks, providers, or IPC here when needed.
}

exports.registerToolPkg = registerToolPkg;

packages/hello.js

/* METADATA
{
  "name": "HelloMira",
  "description": "Simple greeting tools",
  "category": "Utility",
  "tools": [
    {
      "name": "hello_mira",
      "description": "向指定名字发送一句问候",
      "parameters": [
        {
          "name": "name",
          "description": "要问候的名字",
          "type": "string",
          "required": true
        }
      ]
    }
  ]
}
*/

async function hello_mira(params) {
  const name = String(params.name || "朋友");
  complete({ success: true, message: `你好，${name}。` });
}

exports.hello_mira = hello_mira;

toolpkg_id 要长期稳定且全局唯一。已经发布后不要为了换品牌随便改 ID，否则用户会得到两个互不关联的包。

4. 打包、调试与安装

ToolPkg 本质是 ZIP，但不是“执行一次 JS”那么简单。它还要解析清单、安装资源、启用容器、注册子包、刷新 Hook 和运行时映射。

推荐：调试安装脚本

# Windows，设备已开启 USB 调试
tools\debug_toolpkg.bat path\to\hello_mira

# 指定设备
tools\debug_toolpkg.bat path\to\hello_mira --device DEVICE_SERIAL

# Linux / macOS
bash tools/debug_toolpkg.sh path/to/hello_mira

查看日志

adb logcat -d -s ToolPkgDebugInstallReceiver:* ToolPkg:* PackageManager:*
adb logcat -d -s JsEngine:* ToolPkg:* PackageManager:*
  1. 确认脚本返回安装成功，ToolPkg 容器已经启用。
  2. 进入包详情，启用实际子包；只开容器不等于工具可用。
  3. 给当前角色开放目标包与 use_package。
  4. 新建会话，让工具快照重新生成。
  5. 用与工具描述高度一致的提示词测试，并保留原始 Tool Call 轨迹。

5. 发布到 MiraForge

  1. 在 Mira 的 GitHub 账号设置中完成登录。
  2. 进入插件市场或扩展管理的发布入口。
  3. 首次发布时确认创建公开仓库 MiraForge。
  4. 选择脚本或 ToolPkg，填写稳定 ID、版本、说明、分类、源码与权限信息。
  5. Mira 先把文件上传到你自己的 MiraForge，再向兼容市场登记元数据。
  6. 发布后从另一台设备或干净安装状态验证下载、启用、调用和更新。
版本更新不能覆盖事实包版本应使用递增的语义化版本。修改已发布条目时保留同一个稳定 ID、作者仓库和来源链，Release 里不要悄悄替换同名文件。

6. Operit 兼容规则

  • Mira 继续识别既有 JS Bridge、ToolPkg、Skill、MCP 和 .operit/market.json 标记。
  • 现有 OperitForge 条目按原仓库或 Release 下载和更新，不要求重新发布。
  • 新条目优先发布到 MiraForge；需要兼容 Operit 时避免使用只在 Mira 新版本存在的 API。
  • 内部仍可能出现 com.ai.assistance.operit、OPERIT_DOWNLOAD_DIR 等稳定协议名。它们是兼容接口，不是仓库或更新源。
  • 发布说明必须写清最低 Mira/Operit 版本、权限、外部服务、环境变量和已知限制。

7. 安全与隐私检查

  • 只读取完成任务所需的目录、文件和聊天上下文，不扫描无关私人数据。
  • Token 和 API Key 通过环境变量或用户配置传入，不硬编码进脚本、日志或示例。
  • 联网工具明确列出域名、请求字段、超时、重试与数据去向。
  • 文件覆盖、删除、批量操作和高权限调用提供预览、确认和可恢复路径。
  • 所有异常都返回结构化失败结果，不把堆栈、Header、Cookie 或完整路径直接发给模型。
  • UI、后台 Hook 和长任务在卸载或禁用时释放监听器、线程、文件和网络连接。
